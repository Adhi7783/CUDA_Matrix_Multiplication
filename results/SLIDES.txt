SLIDE 3: KERNEL STRATEGIES
================================================================================

Four Kernel Implementations:

[1] NAIVE CUDA
    • Global memory only
    • 1 element per thread
    • No optimization
    • Expected: 60–120 GFLOP/s

[2] TILED (TILE=16)
    • 16×16 shared memory tiles
    • Memory traffic: 16× reduction vs naive
    • Expected: 150–300 GFLOP/s

[3] TILED (TILE=32)
    • 32×32 shared memory tiles
    • Memory traffic: 32× reduction vs naive
    • Better data reuse
    • Expected: 250–380 GFLOP/s

[4] REGISTER-BLOCKED ⭐ WINNER
    • Each thread: 4×4 register block
    • Shared + register optimization
    • Memory traffic: 128× reduction vs naive
    • Expected: 300–450 GFLOP/s

[REFERENCE] CUBLAS
    • NVIDIA cuBLAS (vendor library)
    • Expected: 1,500–2,300 GFLOP/s

SLIDE 9: LIMITATIONS & FUTURE WORK
================================================================================

Why We Don't Reach Peak Performance:

1. INHERENT LIMITATION
   • Matrix multiply is O(N³) work on O(N²) data
   • Compute-to-memory ratio forces memory-bound behavior
   • Even cuBLAS limited by physics (though achieves 95%)

2. MEMORY HIERARCHY BOTTLENECK
   • Current kernels: 14–18% of peak
   • Limited by L2/memory system capacity
   • Potential improvements: 25–35% with advanced techniques

3. PASCAL ARCHITECTURE LIMITATIONS
   • sm_61 lacks some modern features (e.g., tensor cores, async copies)
   • Newer GPUs (Turing sm_75+, Ampere sm_80+) have better memory BW

Recommended Next Steps:
  ✓ Profile memory access patterns (nvprof/nsys)
  ✓ Implement double-buffering (overlap compute + memory I/O)
  ✓ Test mixed-precision variants (FP16 for higher throughput)
  ✓ Benchmark on newer GPU architectures
  ✓ Implement kernel fusion (combine with activation functions)
